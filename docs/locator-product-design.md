# Locator 元素定位产品设计

## 目标用户与核心问题

Locator 面向编写 Selenium Web 自动化脚本的测试和开发人员。用户真正需要的不是“能生成一个选择器”，而是快速得到可复制、可复验、后续维护成本可控的定位代码。

当前高频问题：

- 同一元素可能同时生成 CSS 和 XPath，但用户无法判断哪一个更稳定。
- 页面更新后，历史定位是否仍有效只能手工回到 Console 验证。
- 从选择器到 Selenium 代码仍需手工转换，iframe 和 Shadow DOM 尤其容易写错。
- 连续拾取同一元素会产生重复记录，历史较多时难以查找。

## 产品与架构审查结论

本轮只实现会直接影响正确性或核心操作可达性的项目：

- **必须实现：真实目标归一化。** 跳过 Chrome 翻译插入的重复 `font`，并把按钮内文字提升到实际可操作控件，否则生成结果会稳定失败或定位错误对象。
- **必须实现：完整 iframe 链。** 嵌套 iframe 必须从外到内保存每层定位并生成多次切换代码；只保存最内层 frame 会产生看似可复制、实际不可执行的 Selenium 代码。
- **必须实现：窄面板操作可达。** DevTools 停靠右侧时操作列保持在可视区域，避免用户必须横向滚动到底才能验证、复制或删除。
- **必须实现：诚实反馈。** 没有唯一定位时不能显示“已获取稳定定位”，应明确说明元素已识别但定位不可用。
- **必须实现：多候选稳定性推荐。** 用户需要看到最优、备选、兜底，而不是在 CSS 与 XPath 之间自行猜测；每个候选都必须说明命中依据和评分过程。
- **必须实现：重复条件上下文消歧。** class、文本或属性命中多条时，应结合稳定祖先、同行链接或容器内关联元素生成唯一定位，而不是直接放弃或使用数字位置。

以下方向有价值，但不是当前必须项：

- **Page Object 导出：** 按页面批量生成 Python/Java 类，适合已有大量定位历史的团队；当前单条复制已经覆盖主要工作流。

仍不采用数字位置回退、AI 自愈和完整操作录制器；这些能力会生成表面可用但容易误点的脚本，风险高于收益。

## 本期实现

### 稳定性等级

每个候选定位显示高、中、低或不可用，并说明评级原因。推荐项由 CSS 与 XPath 的质量评分共同决定，不固定优先 CSS。

- 高：`data-testid`、`data-test`、`data-qa`、`data-cy`、稳定 `id`、`name`。
- 中：ARIA 属性、通用稳定 `data-*`、属性组合、稳定 class、相对定位、文本 XPath。
- 低：仅依赖标签，或依赖 `title`、`href`、`type` 等容易变化的弱属性。
- 不可用：没有通过唯一性验证的定位。

依赖页面文案的推荐 XPath 必须明确提示文案变化风险。

### 多候选推荐算法

算法按以下顺序执行，但不会因为某种类型排在前面就跳过其他类型：

1. 为目标元素生成专用测试属性、稳定 `id`、`name`、ARIA、普通属性、class、文本、CSS 和 XPath 候选。
2. 在元素所在的 Document 或 Shadow Root 中执行候选，只有恰好命中一次且命中对象就是用户所选元素时，候选才允许进入推荐列表。
3. 对命中多条的基础条件向上查找最多六层祖先，优先组合稳定祖先；仍不唯一时，在最多 80 个容器后代中寻找唯一链接、属性或短文本作为上下文锚点。
4. 所有组合候选再次执行唯一性验证，排除 `nth-child`、数字 XPath 下标、绝对 XPath 和屏幕坐标。
5. 按基础策略分、上下文依赖扣分、直接定位优先级和选择器长度排序，取前三项展示为“最优、备选、兜底”。

每个候选保存基础选择器、基础命中数量、上下文锚点、扣分、最终分数和稳定性原因。例如基础 class 命中 3 个元素，而当前行内某个链接唯一时，可生成 `//tr[.//a[@href='...']]//sup[...]`，并明确显示“基础命中 3 个，结合同行唯一链接后命中 1 个”。

### 重新验证与高亮

用户可对历史记录点击“验证”。插件在当前页面重新执行推荐定位：

- 仅当匹配结果恰好为一个元素时判定有效。
- 验证成功后在页面高亮该元素约 1.5 秒。
- iframe 按采集时的 frame 上下文过滤；Shadow DOM 按 host 链逐层进入开放的 Shadow Root。
- 嵌套 iframe 按深度和页面 URL 过滤，并保存由外到内的 frame 定位链。
- 验证中、有效、失效均有明确状态，超时按失效处理。

### Selenium 代码

支持复制 Python、Java、JavaScript (`selenium-webdriver`) 代码。生成内容包含：

- CSS 或 XPath 的 `findElement` 调用。
- iframe 的 `switchTo.frame` / `switch_to.frame`。
- 多层 iframe 会按由外到内的顺序生成多次 frame 切换。
- Selenium 4 的 Shadow Root 逐层查找。
- 字符串转义和上下文缺失警告。

### 历史管理

- 可按元素名称、页面 URL、CSS、XPath 搜索。
- 重复拾取相同页面、相同上下文、相同推荐定位时合并记录，并移动到列表顶部。
- 语言偏好只存放在本地；定位历史继续使用浏览器本地扩展存储。

## 明确不做

- 不生成屏幕坐标、`nth-child`、绝对数字 XPath：页面布局或列表顺序变化后极易失效。
- 不做“自愈定位”：自动猜测相似元素可能让测试操作错误对象，却仍显示通过。
- 不做 AI 定位：当前问题可由确定性 DOM 规则解决，引入网络、隐私和不可重复性没有收益。
- 不做完整操作录制器：会扩大权限、性能和维护范围，偏离 Locator 的单一任务。

## 验收标准

- 当唯一标签 CSS 与唯一文本 XPath 同时存在时，文本 XPath 应作为中稳定性推荐项，标签 CSS 只标记为低。
- `data-testid`、稳定 `id` 或 `name` 的唯一定位应评级为高。
- 所有展示为最优、备选、兜底的候选都必须唯一命中用户选择的同一个元素。
- 基础 class 或文本命中 3 个元素时，应能通过稳定祖先或同行关联元素收敛为唯一，并展示基础命中数、上下文和扣分。
- 点击按钮内部文字或图标时，应优先返回可操作的按钮或链接。
- Chrome 翻译插入嵌套 `font[dir="auto"]` 时，应跳过浏览器包装并定位到网站原始元素。
- 验证仅在唯一命中时成功，并能在顶层页面、iframe 和开放 Shadow DOM 中高亮目标。
- 三种语言生成的普通、iframe、Shadow DOM 代码均可直接作为 Selenium 4 脚本片段使用。
- 连续拾取同一定位不会增加历史条数；搜索不会修改或删除原始历史。
- Locator 不读取输入框值、Cookie、请求体或页面存储，数据不发送到第三方服务器。
